**Status:** Complete

## Decisions Made

The following requirements for the meta-canary QR code generator are confirmed:

| Requirement | Decision |
|---|---|
| **Create** | Generate a QR code linked to a URL, quickly and easily |
| **Access control** | Password-protected — not open to the public |
| **Persistence** | Previously generated QR codes are stored and viewable |
| **Export** | QR codes can be downloaded as PNG or SVG files |
| **Cost** | Free to run — no per-use fees |
| **URL encoding** | QR code encodes a URL directly (not a redirect) |

---

## Items Parked

- **QR code redirect approach** — explicitly out of scope for the meta-canary. The simpler direct-encode approach is sufficient for a volume of ~12 codes per year. The redirect approach (where the destination URL can be updated without regenerating the QR code) is a valid future enhancement but is not needed now.

---

## Implications Not Yet Surfaced

- **Persistence mechanism:** "Stored and viewable" implies a database or persistent file store. This is a non-trivial architectural decision for a free, self-hosted tool — it will need to be addressed in the specification session. The simplest approach (flat file or SQLite) should be the default assumption unless there is a reason to use something heavier.
- **Password control:** A single shared password (HTTP basic auth or a simple login form) is likely sufficient for an internal tool used by a small team. Multi-user authentication is not implied by the requirements and should be explicitly parked unless raised.
- **"Free to run":** This constrains the hosting choice. The tool must be deployable on a free tier (e.g., Railway free tier, Render free tier, or self-hosted). This should be confirmed before the build session.
